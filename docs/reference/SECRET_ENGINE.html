What is SecretEngine?

A pragmatic, performance-first game engine for solo developers and small teams

Core Identity

SecretEngine is a mobile-first, Vulkan-based, plugin-driven game engine designed to ship commercial games efficiently. Unlike monolithic engines like Unreal, SecretEngine prioritizes small core size, fast iteration, and explicit performance over feature breadth.

~50MB Core Engine Size
<10s Incremental Build Time
C++20 Core Language
Vulkan Graphics API

🎯 Mobile-First

Designed from day one for mobile GPUs, thermal constraints, and limited memory. Desktop performance is a bonus, not the target.

🔌 Plugin-Driven

Everything except core is a plugin. Renderers, input systems, physics—all replaceable without touching the engine core.

⚡ Performance-First

No hidden allocations, explicit memory management, zero runtime overhead. Performance is a design constraint, not an optimization.

📦 Data-Driven

JSON authoring, binary runtime. No scripting language, no runtime parsing. Data scales, logic ossifies.

Design Philosophy

The principles that guide every decision

Prime Directive

"A small, boring, correct engine beats a clever broken one. Discipline beats features."

  • Single Responsibility: Every module, file, class, and function has exactly ONE reason to change. Boring code is correct code.
  • Core is Sacred: Core defines interfaces and contracts. Plugins define implementations. Core never renders, talks to OS, or assumes platforms.
  • Interfaces Before Implementations: No system exists until its interface exists, is reviewed, and is frozen.
  • Data Over Code: Prefer data tables, configs, and descriptors over hardcoded logic, virtual chains, and inheritance trees.
  • Authoring ≠ Runtime: Humans write JSON. Machines run binary. Runtime never parses authoring formats.
  • Explicit Over Implicit: Explicit lifetimes, ownership, dependencies, and costs. Magic is forbidden.
  • Solo-Founder Reality: One person can maintain it. Features justify themselves via games. Complexity is paid only once.

Scope & Vision

What SecretEngine does (and doesn't do)

What SecretEngine IS

  • C++20/23 core with selective C++23 features
  • Vulkan-only rendering (mobile-first assumptions)
  • Plugin-driven architecture (renderer, input, physics are all plugins)
  • JSON-authored, binary-runtime asset pipeline
  • Asset cooker-centric workflow (watch folders, auto-convert)
  • Web-based level editor (JSON output, no editor runtime)
  • Rust for multiplayer backend and tooling (not engine runtime)

What SecretEngine is NOT

  • Not a visual scripting system or blueprint equivalent
  • Not a general-purpose editor (web tool for authoring only)
  • Not supporting OpenGL or DirectX (Vulkan only)
  • Not embedding scripting languages (Lua, Python, etc.)
  • Not feature-for-feature competing with Unreal
  • Not runtime JSON parsing (binary only at runtime)
  • Not dynamic runtime asset import (cooker required)

Business Reality

You are one person building games to earn money. The engine supports the games, not the other way around. Every feature must justify itself in a shipped game. The engine never grows faster than the games need it to.

Design Principles

Non-negotiable rules (status: FROZEN)

Mobile First

If it runs well on mobile, it will scream on desktop. Design for tilers, minimize overdraw, batch aggressively. Desktop is a bonus, not a target.

Rendering is a Service

Renderer consumes scene data and produces pixels. It knows nothing about gameplay. This allows renderer swapping, headless servers, offline sims.

Instancing is Default

Multiple objects share meshes and materials. Per-instance data is cheap. Drawcalls are expensive. Everything assumes instancing first.

Visibility is Binary

Objects are visible, hidden, culled, or occluded. Visibility is decided early, aggressively, and cheaply. Late decisions are expensive.

LOD is Mandatory

Mesh LODs, texture LODs, and shader complexity scale with distance. Content without LOD strategy is invalid content.

Input Latency is Sacred

Input is sampled early, mapped before simulation, never blocked by UI, never delayed by rendering. Gameplay feel > visuals.

Engine Architecture

Four conceptual layers working in harmony

Game Data

Scenes, assets, configs (binary format)

Plugins

Renderer, Input, Physics, Audio, Networking

Core

Interfaces, lifecycle, data flow, plugin manager

Platform

Android, Windows entry points

Startup Sequence

  1. Platform entry point starts
  2. Core initializes memory, logging, job system
  3. Plugin manager scans available plugins
  4. Plugins are loaded (not activated)
  5. Plugin capabilities are registered
  6. Engine validates required capabilities
  7. Plugins are activated according to config
  8. Initial scene is loaded
  9. Main loop begins

Runtime Loop (Per Frame)

1. Platform events collected 2. Input sampled (raw) 3. Input mapped to actions 4. Simulation step 5. Visibility & culling update 6. Render submission 7. UI submission 8. Platform presentation CRITICAL: Rendering never drives gameplay timing

Key Architectural Guarantees

Core never assumes: a renderer exists, input exists, networking exists
Plugins never: talk to each other directly
All interaction: goes through core-defined interfaces

Critical Plugin Rules

NO DIRECT PLUGIN-TO-PLUGIN CALLS. Communication happens via:
1. Core-mediated capability queries
2. Event system (publish/subscribe)
3. Shared read-only data

Plugins that violate this are rejected immediately.

Entity-Component System

Data-oriented scene representation

What is an Entity?

An Entity is NOT an object. It's just an identifier (handle).

struct Entity { uint32_t id; uint32_t generation; // for handle reuse safety }; // Entity(0,0) is always invalid // Entities are POD (plain old data) // Entities are copyable // Entities have no behavior

What is a Component?

A Component is pure data attached to an entity.

// Components are POD structs struct TransformComponent { Vec3 position; Quat rotation; Vec3 scale; Entity parent; // handle, not pointer }; struct RenderableComponent { MeshHandle mesh; // handle, not pointer MaterialHandle material; uint8_t lod_bias; uint8_t visibility_flags; }; // FORBIDDEN: // ❌ Virtual methods // ❌ Destructors with side effects // ❌ Heap allocations inside components // ❌ References to other components // ❌ Pointers (use handles)

Storage: Structure of Arrays (SoA)

Components are stored in separate arrays per type for cache-friendly iteration.

Scene Memory Layout: TransformComponents: [T0, T1, T2, T3, ...] RenderableComponents: [R0, R1, R2, ...] PhysicsComponents: [P0, P1, ...] Entity-to-Component mapping: sparse set or hash map Why SoA? ✅ Cache-friendly iteration ✅ Easy to add/remove component types ✅ No fragmentation ✅ SIMD-friendly

Component Queries

// Query: "Give me all entities with Transform AND Renderable" Query query { required: [TransformComponent, RenderableComponent], excluded: [DisabledTag] }; // Iterate cache-friendly for (auto [entity, transform, renderable] : world->Query(query)) { // Process only matching entities // Direct array access - fast! }

Performance Assumptions

This data model is designed for:
• 10,000+ entities per scene (mobile)
• 50,000+ entities per scene (desktop)
• Cache-friendly iteration is critical
• Random access is expensive
• Component addition/removal is infrequent

Memory Strategy

Explicit, predictable, measurable

Memory Philosophy

"If allocation cost is unknown → it's wrong."

Forbidden Patterns

  • new / delete in hot paths
  • std::shared_ptr in runtime
  • std::vector growing during gameplay
  • malloc/free directly
  • Hidden allocations (string copies, exception handling)
  • RTTI with dynamic memory

Allowed Allocators

System Allocator

When: Startup, plugin loading, infrequent operations
Backed by: OS allocator
Speed: Slow, thread-safe

Linear Allocator (Arena)

When: Per-frame data, temporary data, command buffers
Strategy: Bump-pointer, reset entire arena
Speed: Very fast, cache-friendly

Pool Allocator

When: Fixed-size objects (entities, components)
Strategy: Fixed blocks, O(1) alloc/free
Speed: Fast, no fragmentation

Memory Budgets (Mobile Mid-Range)

Total Available: ~2GB (after OS) Core Engine: 50MB - Plugin Manager: 5MB - Entity Storage: 20MB - Command Buffers: 10MB - Job System: 5MB Renderer: 200MB - Vulkan Objects: 50MB - Frame Buffers: 100MB - Staging: 50MB Assets: 500MB - Textures: 300MB - Meshes: 100MB - Audio: 50MB Game Logic: 100MB - Entities: 50MB - Gameplay: 50MB Reserved: 150MB (OS thrashing buffer)

Handle System (No Pointers)

// Instead of raw pointers, use handles struct MeshHandle { uint32_t index; uint32_t generation; }; Benefits: ✅ No dangling pointers ✅ Serializable ✅ Cache-friendly ✅ Generation catches use-after-free // All long-lived references use handles Entity parent; // handle, not Entity* MeshHandle mesh; // handle, not Mesh* MaterialHandle material; // handle, not Material*

Development Workflow

From concept to shipped game

1. Design in Web Editor

Create scenes in web-based tool → outputs JSON

2. Asset Cooker Processes

Watches folders, converts JPG→texbin, GLTF→meshbin, JSON→scenebin

3. Engine Loads Binary

Runtime only sees cooked .bin files, never raw assets

4. Test on Device

Fast iteration: scene changes → cooker runs → push to device

Iteration Speed

Asset change → Cooker detects → Binary updated → Hot reload (if applicable) → Testing in < 5 seconds

Asset Pipeline

Authoring → Cooking → Runtime (Implemented: .meshbin v1)

Verified Workflow

Use python tools/cook_mesh.py <input.glb> <output.meshbin> to prepare assets for the engine. The resulting binary is stripped of all non-essential data for instant GPU upload.

Authoring Assets (Never Ship)

  • Scenes: JSON (human-readable hierarchy)
  • Meshes: GLTF, FBX, OBJ
  • Textures: PNG, JPG, HDR, EXR
  • Materials: JSON (shader refs + params)
  • Shaders: GLSL (compiled to SPIR-V)

Cooker Responsibilities

  • Watch asset folders for changes (file watcher)
  • Hash inputs to detect modifications
  • Convert formats: JPG→texbin, GLTF→meshbin, JSON→scenebin
  • Build texture atlases (static env, props, UI)
  • Generate LODs (meshes and textures)
  • Strip unused data (debug info, names)
  • Validate references (all handles must resolve)
  • Emit binary runtime blobs (.bin files)

Runtime Binary Formats

Outputs (never touched by humans): .meshbin - Optimized mesh data (vertices, indices, bounds) .texbin - Compressed textures (BC7, ASTC) + mipmaps .scenebin - Flattened entity/component arrays .shaderbin - Pre-compiled SPIR-V shaders .animbin - Compressed animation curves .matbin - Material parameter blocks

Texture Strategy

Use Case Strategy Reason
Static environment Atlas Reduce drawcalls
Props Atlas Shared materials
UI Atlas Minimize state changes
Characters Individual Flexibility needed
Weapons Individual High detail required
VFX Individual Dynamic updates

Critical Rule

Runtime NEVER parses authoring formats. JSON, GLTF, PNG files NEVER ship to device. Only cooked .bin files. This is non-negotiable for performance.

Vulkan Renderer

Mobile-first, explicit control (Status: OPERATIONAL)

3D Instanced Mesh Support
2D Procedural Overlay Support

Renderer Characteristics

Vulkan Only

No OpenGL, no DirectX. Vulkan 1.2+ required. Mobile GPUs (Adreno, Mali, PowerVR) are primary target.

Forward Rendering

Single-pass forward renderer. Deferred is overkill for mobile. Clustered forward for many lights.

Descriptor Indexing

Bindless textures via VK_EXT_descriptor_indexing. Reduces state changes, enables mega-textures.

No Runtime Compilation

All shaders pre-compiled to SPIR-V at cook time. Zero runtime compilation overhead.

Renderer as a Plugin

The renderer is just another plugin. Core knows nothing about Vulkan.

// Core interface (platform-agnostic) class IRenderer { public: virtual void Submit(RenderableData* data, size_t count) = 0; virtual void Present() = 0; }; // VulkanRenderer plugin implements this class VulkanRenderer : public IPlugin, public IRenderer { // Vulkan implementation details hidden from core VkDevice device_; VkSwapchain swapchain_; VkCommandBuffer cmd_buffers_[3]; // etc. }; // Core never includes vulkan.h // Core never knows GPU state exists

Why This Matters

Renderer can be completely replaced without touching core. Want to add a Metal renderer for iOS? Create MetalRenderer plugin implementing IRenderer. Core doesn't care. This is true modularity.

Input System

Modular, hot-swappable, low-latency

Input Architecture (3 Layers)

Raw Device Layer

Touch, keyboard, gamepad events (platform-specific)

Mapping Layer

Device inputs → Actions (configurable)

Action Layer

Gameplay queries: IsActionPressed("Jump")

Input Timing (Critical)

Frame Timeline: 1. Input sampled EARLY (before simulation) 2. Mapped to actions 3. Simulation consumes actions 4. Rendering uses simulation results NEVER: ❌ Input blocked by UI ❌ Input delayed by rendering ❌ Input sampled late in frame Gameplay feel > visuals

Platform-Specific Plugins

Input is platform-specific, so different plugins per platform:

  • AndroidInput: Touch, accelerometer, gamepad via Android NDK
  • WindowsInput: Keyboard, mouse, gamepad via Win32 API
  • All implement: IInputSystem interface (core doesn't care which)

Networking & Multiplayer

Rust infrastructure, message-based

Key Insight

Networking is infrastructure, not gameplay. The network transmits state and synchronizes events. It does NOT own game rules. Game logic does not depend on network timing.

Rust for Multiplayer Backend

Rust handles the authoritative server and networking infrastructure.

Why Rust?

Safety: No undefined behavior in server code
Performance: Zero-cost abstractions
Isolation: Separate from C++ engine runtime

What Rust Does

• Authoritative server
• Deterministic simulation
• Message routing
• State synchronization
• Telemetry aggregation

Communication

• Message-based API
• Strict boundaries
• No shared memory
• FFI for C++ ↔ Rust
• Think separate processes

Deterministic Simulation

Multiplayer simulation prefers determinism for:

  • Lockstep networking (same input → same output on all clients)
  • Replay systems (re-run inputs to reproduce match)
  • Server-side validation (detect cheating)
  • Bandwidth efficiency (send inputs, not states)

Containment

Not everything must be deterministic—only what matters for multiplayer. Graphics, audio, particle systems can remain non-deterministic. Core gameplay logic (movement, combat, physics) must be deterministic.
